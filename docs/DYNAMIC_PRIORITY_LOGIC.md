## 🎯 核心算法原理

### 设计目标

动态优先级系统旨在解决静态优先级的局限性，通过综合考虑多个动态因素，自动调整计划的优先级，帮助用户更好地管理时间和任务。

### 核心思想

```
静态优先级 = 用户设定的重要度（固定不变）
           ↓
动态优先级 = 基础重要度 + 时间因素 + 进度因素 + 超时惩罚
           ↓
实时反映计划的真实紧迫程度
```

### 四大维度

| 维度           | 权重   | 说明                     | 变化频率       |
| -------------- | ------ | ------------------------ | -------------- |
| **基础重要度** | 40%    | 用户主观判断的重要程度   | 不变           |
| **时间紧迫度** | 35%    | 距离截止日期的天数       | 每天变化       |
| **进度偏离**   | 25%    | 实际进度与预期进度的差距 | 任务完成时变化 |
| **超时加成**   | 0-20分 | 计划或任务超时的惩罚     | 超时后每天变化 |

---

## 📐 计算公式详解

### 总公式

```
动态优先级 = 基础重要度(40%) + 时间紧迫度(35%) + 进度偏离惩罚(25%) + 超时加成(0~20)
```

### 1. 基础重要度 (40%)

```typescript
基础得分 = 用户设定的重要度 × 0.4
```

**示例**：
- 重要度 100 → 基础得分 40
- 重要度 75 → 基础得分 30
- 重要度 50 → 基础得分 20

**设计理由**：
- 保留用户主观判断的权重
- 40% 确保用户意图不会被完全覆盖
- 同时留出 60% 空间给动态因素

---

### 2. 时间紧迫度 (35%)

#### 计算逻辑

```typescript
剩余天数 = Math.ceil((截止日期 - 当前日期) / (1000 * 60 * 60 * 24))

if (剩余天数 <= 0)  → 35分  // 今天到期或已过期
if (剩余天数 <= 1)  → 33分  // 明天到期
if (剩余天数 <= 3)  → 30分  // 3天内
if (剩余天数 <= 7)  → 25分  // 一周内
if (剩余天数 <= 14) → 18分  // 两周内
if (剩余天数 <= 30) → 10分  // 一个月内
if (剩余天数 > 30)  → 线性递减至 0分
```

#### 分段设计原因

```
天数区间     紧迫度评分    心理感受
----------------------------------------
≤ 0天        35分 (100%)  极度紧急
1天          33分 (94%)   非常紧急
2-3天        30分 (86%)   很紧急
4-7天        25分 (71%)   紧急
8-14天       18分 (51%)   有点急
15-30天      10分 (29%)   还有时间
> 30天       递减至0      充裕
```

**设计理由**：
- 非线性递减符合人类对时间的感知
- 最后几天的紧迫感远高于早期阶段
- 分段阈值基于常见时间管理实践（一周、两周、一个月）

---

### 3. 进度偏离惩罚 (25%)

#### 计算逻辑

```typescript
// 步骤1: 计算实际完成进度
已完成任务数 = plan.Tasks.filter(t => t.completed).length
总任务数 = plan.Tasks.length
实际进度 = 已完成任务数 / 总任务数  // 0-1

// 步骤2: 计算预期完成进度
总时长 = 截止日期 - 开始日期
已过时长 = 当前日期 - 开始日期
预期进度 = 已过时长 / 总时长  // 0-1

// 步骤3: 计算进度偏离
进度偏离 = 预期进度 - 实际进度

// 步骤4: 转换为惩罚分数
if (进度偏离 <= 0) {
  // 超前或符合预期，不惩罚
  进度惩罚 = 0
} else {
  // 落后，线性惩罚
  进度惩罚 = 进度偏离 × 25
}

// 步骤5: 超时任务额外惩罚
超时任务数 = plan.Tasks.filter(t => !t.completed && t.date < now).length
if (超时任务数 > 0) {
  进度惩罚 += Math.min(10, 超时任务数 × 2)
}

// 最终惩罚分数
进度惩罚 = Math.min(35, 进度惩罚)  // 最高35分
```

#### 示例计算

**案例1：按计划进行**
```
计划周期: 1月1日 - 1月31日 (30天)
当前日期: 1月16日 (已过15天，50%时间)
总任务: 10个
已完成: 5个

预期进度 = 15/30 = 0.5 (50%)
实际进度 = 5/10 = 0.5 (50%)
进度偏离 = 0.5 - 0.5 = 0
进度惩罚 = 0分 ✅ 按计划进行
```

**案例2：进度落后**
```
计划周期: 1月1日 - 1月31日 (30天)
当前日期: 1月16日 (已过15天，50%时间)
总任务: 10个
已完成: 3个

预期进度 = 15/30 = 0.5 (50%)
实际进度 = 3/10 = 0.3 (30%)
进度偏离 = 0.5 - 0.3 = 0.2
进度惩罚 = 0.2 × 25 = 5分 ⚠️ 轻微落后
```

**案例3：严重落后 + 超时任务**
```
计划周期: 1月1日 - 1月31日 (30天)
当前日期: 1月26日 (已过25天，83%时间)
总任务: 10个
已完成: 4个
超时未完成任务: 3个

预期进度 = 25/30 = 0.83 (83%)
实际进度 = 4/10 = 0.4 (40%)
进度偏离 = 0.83 - 0.4 = 0.43
基础惩罚 = 0.43 × 25 = 10.75分
超时惩罚 = min(10, 3 × 2) = 6分
总惩罚 = 10.75 + 6 = 16.75分 ❌ 严重落后
```

**设计理由**：
- 奖励超前完成（0惩罚）
- 惩罚进度落后（比例惩罚）
- 超时任务额外惩罚（强调deadline重要性）

### 4. 超时处理加成 (0-20分)

#### 计算逻辑

```typescript
超时加成 = 计划整体超时加成 + 超时任务加成

// 1. 计划整体超时加成（最高15分）
if (当前日期 > 截止日期 && !已完成) {
  超时天数 = 当前日期 - 截止日期
  计划超时加成 = Math.min(15, 超时天数 × 2)
}

// 2. 超时任务加成（最高10分）
超时任务数 = plan.Tasks.filter(t => !t.completed && t.date < 当前日期).length
if (超时任务数 > 0) {
  任务超时加成 = Math.min(10, 超时任务数 × 3)
}

// 总加成
超时加成 = Math.min(20, 计划超时加成 + 任务超时加成)
```

#### 示例

**案例1：计划超时2天**
```
计划截止: 1月15日
当前日期: 1月17日
超时天数 = 2天

计划超时加成 = min(15, 2 × 2) = 4分
```

**案例2：3个任务超时**
```
超时任务数 = 3个

任务超时加成 = min(10, 3 × 3) = 9分
```

**案例3：计划超时 + 任务超时**
```
计划超时: 5天 → 计划加成 = min(15, 5 × 2) = 10分
超时任务: 4个 → 任务加成 = min(10, 4 × 3) = 10分
总加成 = min(20, 10 + 10) = 20分 ⚠️ 最高加成
```

## 📊 排序规则

### 三级排序逻辑

```typescript
排序优先级：
1. 默认计划（未分类计划）→ 永远置顶
2. 动态优先级 → 高到低排序
3. 截止日期 → 未截止的在前
```

### 实现代码

```typescript
const sorted = plansWithPriority.sort((a, b) => {
  // 第一级：默认计划置顶
  const isADefault = isDefaultPlan(a.id);
  const isBDefault = isDefaultPlan(b.id);
  if (isADefault && !isBDefault) return -1;
  if (!isADefault && isBDefault) return 1;
  
  // 第二级：按动态优先级降序
  if (a.dynamicPriority !== b.dynamicPriority) {
    return b.dynamicPriority - a.dynamicPriority;
  }
  
  // 第三级：优先级相同时，未截止的在前
  const now = new Date();
  const aOverdue = a.dueDate && new Date(a.dueDate) < now;
  const bOverdue = b.dueDate && new Date(b.dueDate) < now;
  
  if (aOverdue && !bOverdue) return 1;  // a超时，b未超时，b在前
  if (!aOverdue && bOverdue) return -1; // a未超时，b超时，a在前
  
  return 0; // 都超时或都未超时，保持相对顺序
});
```

### 排序示例

```
输入计划列表:
- 未分类计划: 动态优先级 0, 未截止
- 考试准备: 动态优先级 95, 已截止
- 项目汇报: 动态优先级 95, 未截止
- 学习Python: 动态优先级 65, 未截止
- 健身计划: 动态优先级 45, 已截止
- 读书笔记: 动态优先级 45, 未截止

排序后:
1. 未分类计划 (置顶规则)
2. 项目汇报 (优先级95, 未截止) ← 同优先级，未截止优先
3. 考试准备 (优先级95, 已截止)
4. 学习Python (优先级65, 未截止)
5. 读书笔记 (优先级45, 未截止) ← 同优先级，未截止优先
6. 健身计划 (优先级45, 已截止)
```

## ⏰ 超时判定逻辑

### 关键问题

**问题**：今天（1月17日）下派的任务，设定日期为今天（1月17日），是否应该立即显示为"超时"？

**答案**：❌ 不应该。只有到了第二天（1月18日）且任务未完成，才算超时。

### 正确的超时判定

```typescript
function isTaskOverdue(task: Task): boolean {
  const now = new Date();
  const taskDate = new Date(task.date);
  
  // 关键：设置时间为当天的 23:59:59
  taskDate.setHours(23, 59, 59, 999);
  
  // 只有当前时间超过任务日期的当天结束时间，且任务未完成，才算超时
  return !task.completed && now > taskDate;
}
```

### 时间线示例

```
场景：今天是 1月17日 15:00

任务A: 日期设为 1月16日
判定: now (1/17 15:00) > taskDate (1/16 23:59:59) ✅ 超时

任务B: 日期设为 1月17日
判定: now (1/17 15:00) > taskDate (1/17 23:59:59) ❌ 未超时

任务C: 日期设为 1月18日
判定: now (1/17 15:00) > taskDate (1/18 23:59:59) ❌ 未超时

---

第二天：1月18日 00:01

任务B: 日期设为 1月17日
判定: now (1/18 00:01) > taskDate (1/17 23:59:59) ✅ 超时（凌晨后）
```

#### 在 `priority.ts` 中

```typescript
/**
 * 判断任务是否超时
 * 只有过了任务日期的当天 23:59:59，且任务未完成，才算超时
 */
function isTaskOverdue(task: Task, now: Date = new Date()): boolean {
  if (task.completed) return false;
  
  const taskDate = new Date(task.date);
  // 设置为当天的最后一刻
  taskDate.setHours(23, 59, 59, 999);
  
  return now > taskDate;
}

/**
 * 计算进度偏离惩罚
 */
function calculateProgressPenalty(plan: Plan): number {
  const now = new Date();
  
  // ... 其他代码 ...
  
  // 使用新的超时判定逻辑
  const overdueTasks = plan.Tasks.filter(t => isTaskOverdue(t, now));
  
  if (overdueTasks.length > 0) {
    penalty += Math.min(10, overdueTasks.length * 2);
  }
  
  return Math.min(35, penalty);
}

/**
 * 检查是否有超时任务
 */
export function hasOverdueTasks(plan: Plan): boolean {
  const now = new Date();
  return plan.Tasks.some(t => isTaskOverdue(t, now));
}

/**
 * 获取需要重新安排的任务列表
 */
export function getTasksNeedReschedule(plan: Plan): Array<{
  taskId: string;
  taskName: string;
  originalDate: Date;
  isStrictMode: boolean;
}> {
  const now = new Date();
  const strict = isStrictMode(plan);
  
  // 使用新的超时判定逻辑
  const overdueTasks = plan.Tasks.filter(t => isTaskOverdue(t, now));
  
  return overdueTasks.map(task => ({
    taskId: task.id,
    taskName: task.name,
    originalDate: task.date,
    isStrictMode: strict
  }));
}
```

## 🔍 实现细节

### 数据流转

```
1. 用户创建计划 (AIPlanner)
   ↓
   设置基础重要度 (importance)
   ↓
2. 存储到数据库
   ↓
   Plans 表: priority 字段
   ↓
3. 加载到 Store (taskStore)
   ↓
   Plans 数组
   ↓
4. 实时计算动态优先级 (priority.ts)
   ↓
   getDynamicPriority(planId)
   ↓
5. UI 显示 (PlanList)
   ↓
   按动态优先级排序和显示
```

### 计算时机

```
触发计算的时机:
✅ 1. 页面加载时 (useEffect)
✅ 2. Plans 数据变化时
✅ 3. 任务完成状态改变时
✅ 4. 每次进入 PlanView 时

不需要触发的情况:
❌ 1. 时间自然流逝（下次进入页面时会重新计算）
❌ 2. 编辑计划名称等无关信息时
```

## ⚠️ 边界情况处理

### 1. 无截止日期的计划

```typescript
if (!plan.dueDate) {
  urgencyScore = 0;  // 时间紧迫度为0
  overtimeBonus = 0; // 超时加成为0
}
```

### 2. 无任务的计划

```typescript
if (plan.Tasks.length === 0) {
  progressPenalty = 0; // 进度惩罚为0
}
```

### 3. 已完成的计划

```typescript
if (plan.completed) {
  return 0; // 动态优先级直接返回0
}
```

### 4. 默认计划（未分类计划）

```typescript
if (isDefaultPlan(planId)) {
  return 0; // 不使用动态优先级，固定为0
}
```

### 5. 开始日期晚于当前日期

```typescript
if (plan.startDate > now) {
  expectedProgress = 0; // 预期进度为0
  progressPenalty = 0;  // 进度惩罚为0
}
```

### 6. 计划周期为0或负数

```typescript
const totalDuration = plan.dueDate - plan.startDate;
if (totalDuration <= 0) {
  progressPenalty = 0; // 无法计算进度，惩罚为0
}
```

## 🎯 设计原则总结

### 1. 平衡性
- 用户意图（40%）与系统判断（60%）平衡
- 时间因素（35%）与进度因素（25%）平衡

### 2. 公平性
- 按计划完成的不惩罚
- 超前完成的不额外奖励（避免博弈）
- 落后的线性惩罚（公平透明）

### 3. 激励性
- 临近截止时优先级自动提升
- 落后时给予警示（高优先级）
- 完成后优先级归零（鼓励完成）

### 4. 可预测性
- 公式透明，计算逻辑清晰
- 分段阈值基于常识（一周、两周等）
- 线性关系为主，便于理解

### 5. 实用性
- 默认计划特殊处理（不干扰日常零散任务）
- 严格/宽松模式（适应不同场景）
- 超时判定合理（不提前惩罚）
