# 动态优先级实现说明

## 目的

根据静态重要度和任务执行情况动态调整计划优先级，便于排序和提醒。

## 总体公式

动态优先级 = 基础重要度(40%) + 时间紧迫度(35%) + 进度偏离惩罚(25%) + 超时加成(0–20)

总分通常在 0–100 范围内，达到 0 表示无需关注，越高越紧迫。

## 各部分说明

1. 基础重要度（40%）

- 由用户设定的重要度（0–100）直接按 40% 权重计算：基础得分 = 重要度 × 0.4。

2. 时间紧迫度（35%）

- 根据距离截止日期的天数映射到分值段：
  - <=0 天（当天或已过）：35
  - 1 天：33
  - <=3 天：30
  - <=7 天：25
  - <=14 天：18
  - <=30 天：10
  - > 30 天：按线性方式递减到 0
- 目的：靠近截止日的计划优先级显著上升。

3. 进度偏离惩罚（25%）

- 定义：
  - 实际进度 = 已完成任务数 / 总任务数
  - 预期进度 = 已过时长 / 总时长
  - 进度偏离 = 预期进度 - 实际进度（若 <=0，则不惩罚）
- 计算：
  - 基础惩罚 = 进度偏离 × 25
  - 若存在超时未完成任务，再加 min(10, 超时任务数 × 2)
  - 最终进度惩罚上限 35 分
- 说明：落后会被惩罚，超前或按计划则不惩罚。

4. 超时加成（0–20）

- 计划整体超时（最高 15 分）：若当前日期 > 截止日期且计划未完成，按超时天数 × 2，取上限 15。
- 单任务超时加成（最高 10 分）：超时未完成任务数 × 3，取上限 10。
- 总加成取 min(20, 计划超时加成 + 任务超时加成)。

## 排序规则

1. 默认计划（未分类）始终置顶。
2. 按动态优先级降序排序。
3. 若优先级相同，未截止的在前，已截止的在后。

## 超时判定

- 当日任务不视为超时。判断方法：将任务日期设为当天 23:59:59，只有当前时间超过该值且任务未完成才算超时。

示例判定函数（伪代码）：

```typescript
function isTaskOverdue(task: Task, now = new Date()) {
  if (task.completed) return false;
  const d = new Date(task.date);
  d.setHours(23, 59, 59, 999);
  return now > d;
}
```

## 实现要点

- 在渲染或排序时按需计算动态优先级，不必频繁实时重算。
- 触发时机：页面加载、Plans 或任务状态变化、进入计划详情页。
- 使用 useMemo 缓存排序结果以提升性能。
- 在进度与超时计算中处理边界情况（无截止日期、无任务、开始日期晚于当前、周期为 0 等）。

## 边界情况

- 无截止日期：时间紧迫度和超时加成为 0。
- 无任务：进度惩罚为 0。
- 已完成计划：优先级返回 0。
- 开始日期在未来：预期进度为 0，不惩罚。

## 建议

- 保留可配置权重以适配不同用户需求。
- 将频繁计算的部分（如排序）加缓存或节流。
- 在 store 层统一计算并暴露给 UI 使用，避免组件重复实现。

## 示例（简要）

- 重要度 80、剩余 3 天、按计划：动态优先级 ≈ 80×0.4 + 30 + 0 = 62
- 重要度 60、剩余 15 天、落后较多且有超时任务：按公式合并计算得到高优先级。
